> 这篇笔记以 LangChain 当前官方文档为准。白皮书中的“模型、工具、编排层”和“厨师做菜”类比，用来帮助理解，不作为 LangChain API 的定义。

## 先抓住一句话

LangChain 对 Agent 的定义很直接：**Agent 是一个让模型在循环中调用工具，直到任务完成的系统。**

官方还给出一个更适合记忆的表达：

```text
Agent = Model + Harness
```

这里的 `Harness` 可以理解为包在模型循环外面的**运行与编排系统**。它负责在正确的时机，把正确的上下文、提示词和工具交给模型，并控制循环如何继续、暂停或结束。

所以，Agent 不是“一个更聪明的模型”，而是**模型被放进一个能够判断、行动、接收反馈并继续执行的系统里**。

## 从“回答问题”到“完成任务”

单独调用模型时，常见过程是：

```text
用户输入 -> 模型生成 -> 返回答案
```

Agent 的过程则可能是：

```text
用户给出目标
-> 模型判断下一步
-> 调用工具
-> 读取工具结果
-> 更新状态
-> 再次判断
-> 任务完成或触发停止条件
```

关键变化不只是“可以调用工具”，而是**模型能够根据中间结果动态决定下一步**。执行路径不一定在开始时就完全写死。

例如，用户要求“查询上海天气并给出出行建议”时，Agent 可能先调用天气工具，再根据返回的温度和降雨信息组织建议。如果工具失败，Harness 还可以通过重试、降级或人工审批来处理。

## Chatbot、Workflow 与 Agent 的边界

**Chatbot 偏对话，Workflow 偏确定流程，Agent 偏动态决策和工具调用。** 这是区分三者时最实用的第一层判断。

不过，三者并不完全处在同一维度：Chatbot 更像用户看到的**交互形式**，Workflow 和 Agent 更像系统内部的**任务执行方式**。一个 Chatbot 背后既可以只有一次模型调用，也可以连接 Workflow，甚至可以把 Agent 包装成聊天界面。

| 对比维度 | Chatbot | Workflow | Agent |
|---|---|---|---|
| 核心目标 | 持续理解并回应用户 | 按预先规定的步骤完成任务 | 围绕目标动态选择下一步 |
| 主要控制者 | 对话逻辑和模型 | 开发者定义的流程、分支和规则 | 模型在 Harness 约束内进行运行时决策 |
| 执行路径 | 通常以一问一答为主 | 大部分路径在运行前已经确定 | 路径会随工具结果和当前状态变化 |
| 工具使用 | 可有可无 | 通常在固定节点调用固定工具 | 模型可以根据情况选择工具、参数和调用时机 |
| 适合任务 | 问答、解释、陪伴式交互 | 审批、数据处理、固定业务流程 | 调研、排障、复杂助理等开放任务 |
| 主要代价 | 容易只有“会说”而不能行动 | 稳定但不擅长处理流程外情况 | 成本、延迟和不确定性更高，需要权限与停止规则 |

![Chatbot、Workflow 与 Agent 的边界](<notes_pictures/06-chatbot-workflow-agent-boundaries.png>)

可以用三个例子建立直觉：

- **Chatbot**：用户询问“公司的请假规则是什么？”，系统围绕问题进行解释和追问。
- **Workflow**：用户提交请假申请后，系统固定执行“校验表单 -> 发送审批 -> 更新状态 -> 通知用户”。
- **Agent**：用户提出“帮我安排下周去上海出差”，系统需要根据中间结果动态查询交通、比较方案、检查日历，再决定是否继续调用其他工具。

一个实用的判断顺序是：

```text
主要价值是自然语言对话？
-> Chatbot

步骤能否在运行前基本写清楚？
-> 能：Workflow
-> 不能，下一步取决于中间结果，并需要模型反复选择工具：Agent
```

容易混淆的地方是：

- **能聊天不等于 Agent**：普通 Chatbot 可能只是把历史消息反复传给模型。
- **用了模型不等于 Agent**：Workflow 可以在固定节点调用模型做分类、摘要或生成。
- **调用一次工具不一定是 Agent**：关键在于工具结果是否返回循环，让模型继续决定下一步。
- **Agent 不一定必须是聊天产品**：它也可以由定时任务、API 请求或系统事件触发。

实际应用经常是混合结构：**Chatbot 负责交互，Workflow 固定关键业务边界，Agent 只处理其中需要动态判断的部分。**

## 用“厨师做菜”理解 Agent

白皮书把 Agent 类比成厨师，这个类比很适合建立直觉：

| 厨房里的角色或动作 | Agent 中的对应物 |
|---|---|
| 顾客点菜 | 用户目标 |
| 菜谱知识与现场判断 | Model |
| 刀具、灶台、食材 | Tools |
| 后厨流程和操作规范 | Harness / 编排层 |
| 查看食材、尝味道 | 获取工具结果与环境反馈 |
| 调整做法，继续烹饪 | 更新状态后进入下一轮 |
| 菜品完成 | 达到目标，输出最终结果 |

厨师不是只想一次就结束。他会接收目标、检查条件、选择工具、执行动作，再根据结果调整。Agent 也是如此：**判断和行动交替进行，反馈又成为下一轮判断的上下文。**

## Agent 的核心结构

白皮书用 `Model + Tools + Orchestration` 描述 Agent；LangChain 当前文档用 `Agent = Model + Harness` 描述。两者观察角度不同，但可以放在同一张心智图里理解。

### Model：负责判断

模型读取当前上下文，判断应该直接回答，还是调用某个工具，以及工具需要哪些参数。

模型负责的是**选择下一步行动**，并不亲自完成搜索网页、查询数据库或写文件等外部操作。

### Tools：负责行动

工具是模型连接外部世界的接口，可以是 Python 函数、LangChain Tool 或工具描述，例如：

- 查询天气或搜索资料
- 读取数据库
- 运行代码
- 读写文件
- 调用业务 API

工具扩展了 Agent 的行动范围，但也扩大了风险范围。一个工具是否可用、参数如何定义、权限有多大，都会直接影响 Agent 的可靠性。

### Harness：负责让循环安全地运行

Harness 不是单个对象的简单别名，而是围绕 Agent 循环的一组运行机制。LangChain 的 `create_agent` 会把模型、工具、系统提示词和中间件组合成一个可执行的 Harness。

它通常负责：

- 准备模型每一轮需要的上下文
- 维护消息和状态
- 执行工具调用并把结果写回循环
- 通过中间件实现重试、调用次数限制和上下文压缩
- 加入护栏、权限控制和 Human-in-the-loop
- 判断何时继续、暂停或结束

因此，`Agent = Model + Harness` 不要机械理解成两个完全分离的零件。更准确的理解是：**模型提供决策能力，Harness 把模型、工具和运行规则组织成一个能持续工作的系统。**

## Agent 的核心循环

一个典型的 Agent 循环可以拆成下面几步：

1. **接收目标**：用户通过消息提出任务。
2. **准备上下文**：Harness 组合消息、系统提示词、工具说明、状态和必要的运行信息。
3. **模型判断**：模型决定直接返回答案，还是产生 Tool Call。
4. **执行工具**：系统校验参数并调用对应工具。
5. **写回结果**：工具结果作为新消息或状态进入上下文。
6. **继续循环**：模型读取新结果，再判断下一步。
7. **结束任务**：模型给出最终答案，或系统触发停止条件、调用上限、人工审批等控制点。

这和 ReAct 的“判断/行动/观察”节奏很相似，但二者不能画等号。**ReAct 是一种帮助模型决定下一步的推理与行动模式；Agent 是包含模型、工具、状态和运行控制的完整系统。** 一个 Agent 可以借鉴 ReAct，也可以使用其他规划和编排方式。

## Messages、State 与 Context

Agent 要多轮执行，就必须知道“现在进行到哪里了”。在 LangChain 中，所有 Agent 的状态都包含一组 `messages`，里面会逐步积累用户消息、模型消息和工具结果。

需要区分三个容易混淆的概念：

- **Messages**：对话和工具交互的事件序列，是 Agent 状态的一部分。
- **State**：一次任务在当前线程中的可变数据，包括消息以及自定义字段。
- **Context**：某次调用传入、供工具和中间件读取的运行信息，例如用户 ID、权限或功能开关；它不等于对话状态。

如果希望本地 Agent 跨多次调用保留并恢复会话历史，只有 `thread_id` 还不够，还需要给 Agent 配置 `checkpointer`。`thread_id` 用来标识同一个会话，`checkpointer` 负责真正保存状态。

## Model 与 Agent 的区别

| 对比维度 | Model | Agent |
|---|---|---|
| 基本角色 | 根据输入生成输出 | 围绕目标持续判断和行动 |
| 外部信息 | 默认依赖训练知识和传入上下文 | 可以通过工具读取实时或私有数据 |
| 行动能力 | 本身不能真正操作外部系统 | 可以通过受控工具影响外部系统 |
| 状态 | 单次调用本身不自动持久化任务状态 | Harness 可以维护消息、状态和检查点 |
| 执行路径 | 通常是一次模型调用 | 可能经过多轮模型与工具交互 |
| 结束方式 | 生成一次输出后结束 | 达到目标或触发停止条件后结束 |

这里说 Model “没有状态”，指的是**一次模型 API 调用本身不会自动替你维护应用状态**。开发者仍然可以把历史消息重新传给模型；真正的状态管理发生在模型外部。

## Tool Calling 不一定等于 Agent

模型能生成一个 Tool Call，只说明它具备**请求调用工具**的能力，还不能单独证明整个应用就是 Agent。

如果程序只让模型选择一次工具，执行后立刻结束，它更像“带工具的一次模型调用”。当系统会把工具结果重新交给模型，并允许模型继续判断、继续行动，直到满足目标或停止条件时，才更符合 LangChain 当前的 Agent 定义。

可以这样记：

```text
Tool Calling 是一种能力
Agent Loop 是一种运行方式
Harness 是承载这种运行方式的系统
```

## Agent 的自主性是有边界的

白皮书强调 Agent 的自主能力，但工程实现中不应把“自主”理解成“没有限制”。LangChain 当前文档也明确指出，完全自主并不总是合适。

更准确的说法是：**Agent 在开发者给定的目标、工具、权限、上下文、调用上限和审批规则内，自主选择下一步。**

高风险动作——例如写文件、发邮件、付款、删除数据或昂贵的 API 调用——应该由确定性的权限规则或 Human-in-the-loop 控制，而不是只在系统提示词里写一句“请谨慎”。

## 在 LangChain 中创建最小 Agent

```python
from langchain.agents import create_agent
from langchain.tools import tool


@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气。"""
    return f"{city}：晴，25℃"


agent = create_agent(
    model="openai:gpt-5.5",
    tools=[get_weather],
    system_prompt="你是一个简洁、准确的出行助手。",
)

result = agent.invoke(
    {
        "messages": [
            {"role": "user", "content": "查询上海天气并给出出行建议"}
        ]
    }
)

print(result["messages"][-1].content)
```

这段代码里：

- `model` 提供判断能力。
- `tools` 提供可执行动作。
- `system_prompt` 约束任务方式。
- `create_agent` 创建并配置 Harness。
- `invoke` 向 Agent 的状态追加用户消息并启动循环。

这个天气工具只是本地演示，没有调用真实天气服务。实际学习时，可以先观察返回的 `messages`，找出模型产生的工具调用、工具结果和最终回答分别对应哪条消息。

## 什么时候不需要 Agent

不是所有 LLM 应用都应该做成 Agent。

- 只需要改写、分类或摘要一次文本：直接调用 Model 通常更简单。
- 步骤固定、顺序明确、每一步都可预先定义：优先考虑确定性 Workflow。
- 下一步必须根据中间结果动态选择，并且需要多轮工具交互：Agent 更合适。

Agent 带来的灵活性，也意味着更高的延迟、成本、不确定性和调试难度。**能用固定流程解决的问题，不必为了“智能”而强行加入 Agent 循环。**

## 最后形成的心智模型

```text
目标
  + 模型判断
  + 工具行动
  + 状态反馈
  + Harness 约束
  = 在边界内持续执行，直到任务结束
```
