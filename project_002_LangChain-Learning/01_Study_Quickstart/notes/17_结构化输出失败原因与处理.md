## 核心判断

**结构化输出失败时，先判断失败发生在哪一层，再决定是改 Schema、改策略、重试模型，还是做业务兜底。**

不要把所有失败都归因于“模型不听话”。结构化输出链路至少包含 Schema 定义、Provider 能力、模型生成、LangChain 解析、本地类型校验、业务规则校验和状态保存。不同层级的失败需要不同处理方式。

## 常见失败原因

| 失败原因 | 常见位置 | 典型表现 | 处理方式 |
| --- | --- | --- | --- |
| Schema 不受支持 | Provider 请求阶段 | 请求直接报错，提示某些 JSON Schema 特性不支持 | 简化 Schema，避免不支持的关键字，确认 strict 模式要求 |
| 字段缺失或类型错误 | LangChain 解析阶段 | 缺少必填字段、字符串变数组、枚举值不匹配 | ToolStrategy 可反馈验证错误并有限重试；ProviderStrategy 需要检查 Schema 和模型能力 |
| 一次提交多个结构化结果 | ToolStrategy | 模型同时调用多个结构化输出工具或返回多个候选结构 | 要求模型只提交一个最终结果，必要时调整 `handle_errors` |
| Provider 原生能力不可用 | Provider 或网关层 | 聊天可用，但 `json_schema`、`strict` 或工具并用失败 | 改用 ToolStrategy，或退回 JSON Mode 加本地校验 |
| 安全拒答 | 模型响应阶段 | 返回 `refusal` 或拒绝内容，而不是目标 Schema | 把拒答作为单独分支处理，不应当反复用 Schema 错误重试 |
| token 截断 | 模型响应阶段 | JSON 不完整、字段缺失、结束原因显示长度限制 | 提高输出上限，简化 Schema，缩短上下文 |
| Prompt 与 Schema 冲突 | 模型生成阶段 | Prompt 要求自然语言解释，Schema 要求固定字段 | 统一系统提示词、字段描述和输出目标 |
| 结构正确但语义错误 | 业务校验阶段 | 字段类型正确，但城市、金额、日期或工具结果不一致 | 增加业务校验、交叉检查和数据源回查 |
| checkpoint 反序列化警告 | 状态保存阶段 | 出现未注册类型反序列化警告 | 区分安全警告和结构化输出失败；生产环境升级依赖并配置 allowlist |

## ToolStrategy 的失败处理

ToolStrategy 的主要优势是可以把结构化输出错误作为工具消息反馈给模型，让模型修正后再次提交。

默认写法：

```python
response_format=ToolStrategy(WeatherResponseFormat)
```

默认 `handle_errors=True`，表示 LangChain 会处理结构化输出相关错误，并尝试让模型修复。

如果希望直接暴露错误，可以关闭自动处理：

```python
response_format=ToolStrategy(
    schema=WeatherResponseFormat,
    handle_errors=False,
)
```

如果希望给模型一个固定修复提示，可以传字符串：

```python
response_format=ToolStrategy(
    schema=WeatherResponseFormat,
    handle_errors="结构化输出不符合 Schema。请只提交一个结果，并修正字段类型。",
)
```

如果希望按异常类型生成不同反馈，可以传回调函数：

```python
def handle_structured_error(error: Exception) -> str:
    return f"结构化输出校验失败：{error}。请根据错误信息修正字段。"


response_format=ToolStrategy(
    schema=WeatherResponseFormat,
    handle_errors=handle_structured_error,
)
```

需要注意：自动修复不应被当成无限重试机制。生产代码仍然需要限制 Agent 最大步数、记录原始响应和最终失败原因。

## ProviderStrategy 的失败处理

ProviderStrategy 的失败更常见于请求阶段或 Provider 原生结构化输出阶段。

典型写法：

```python
response_format=ProviderStrategy(
    schema=WeatherResponseFormat,
    strict=True,
)
```

排查时优先检查：

- 目标模型是否支持原生 Structured Outputs。
- 当前 `base_url` 是否完整实现了对应 Provider 的结构化输出协议。
- Schema 是否使用了 Provider 不支持的 JSON Schema 特性。
- 对象字段是否满足 strict 模式要求，例如明确 required 和禁止额外字段。
- 业务工具和原生结构化输出能否在同一个 Agent 中同时使用。

ProviderStrategy 不可用时，不要只更换类名反复重试。更稳妥的处理是先改成 `ToolStrategy(WeatherResponseFormat)` 验证工具调用路线；如果工具调用也不稳定，再退回 JSON Mode 或 Prompt 约束加本地校验。

## 当前项目的排查顺序

当前项目使用 `ChatOpenAI` 接入 OpenAI-compatible 模型服务，并在 `create_agent()` 中传入 `WeatherResponseFormat`。

建议按下面顺序排查：

1. **确认响应中是否存在 `structured_response`。** 如果存在，先说明结构化解析已经成功。
2. **查看 `messages` 中是否出现结构化输出工具调用。** 如果出现，说明大概率走工具调用路线。
3. **查看 `AIMessage.tool_calls[].args`。** 这里是模型提交的结构化字段。
4. **查看 `ToolMessage`。** 业务工具结果和结构化输出确认消息要分开看。
5. **查看 `finish_reason`。** `tool_calls` 表示本次模型调用以工具调用结束，不等于 Agent 失败。
6. **查看 `usage_metadata` 和本地调用记录。** 判断是否存在 token 截断、重试或多次模型调用。
7. **切换显式策略做对照。** 用 `ToolStrategy(...)` 验证工具调用路线，用 `ProviderStrategy(..., strict=True)` 验证 Provider 原生能力。

## 修复原则

- **先修 Schema，再调 Prompt。** 字段类型、必填、可空、默认值和额外字段策略不清楚时，Prompt 很难稳定补救。
- **先区分格式错误和语义错误。** Schema 只能处理结构，事实正确性需要业务校验。
- **先确认能力，再选择策略。** ProviderStrategy 依赖原生结构化输出；ToolStrategy 依赖工具调用。
- **保留错误证据。** 记录原始响应、验证错误、重试次数、模型名、Provider 和调用参数。
- **设置重试边界。** 自动修复应有限次数，失败后返回可诊断错误，而不是静默吞掉异常。
