## Prompt 调优

找到好的 prompt 不是一次写对，而是一个持续迭代的过程。

可以先从简单 prompt 开始，再逐步加入角色、上下文、例子、输出格式等要素。不要一开始就把所有要求堆进去，否则很难判断到底是哪一部分影响了结果。

如果任务很大，可以先拆成几个小任务分别调试。prompt 调优的本质有点像写代码：先跑通最小版本，再一点点补约束、补上下文、补边界。

## 从训练数据角度理解 Prompt

如果知道模型训练数据大概长什么样，参考训练数据的表达方式来构造 prompt 往往更有效。

比如把 AI 当人来看：你知道 ta 爱读《西游记》，那就更容易用《西游记》里的语言和结构跟 ta 沟通。模型也是类似的，它更容易沿着自己熟悉的文本模式继续生成。

不知道训练数据怎么办？可以看模型是否“主动告诉你”它偏好的格式。比如 OpenAI 的 GPT 系列通常对 Markdown 结构比较友好，清晰标题、列表、代码块、分隔符都能帮助它理解任务边界。

还有一种方式就是不断尝试。有时一字之差，对 token 生成概率的影响可能很大；也可能几乎没有影响。这也是 prompt 工程有点玄学感的来源：门槛低，但天花板很高。

## 指令要具体

高质量 prompt 的核心是：**指令具体、信息丰富、尽量少歧义**。

不要只说：

```text
帮我优化一下这段内容。
```

更好的写法是：

```text
请把下面这段内容改写成适合初学者复习的 Obsidian 笔记。
要求：
- 保留核心概念
- 减少口号式表达
- 用自然段解释直觉和误区
- 不要写成课程讲义
```

**具体不是越长越好，而是相关信息要足够清楚。无关细节太多，也会稀释模型注意力**。

## 使用清晰分隔符

当 prompt 同时包含指令、上下文和输入时，最好用明显的分隔符把它们区分开。

比如：

```text
### 指令 ###
请提取下面文本中的地名。

### 输出格式 ###
地点：<逗号分隔的地点列表>

### 输入文本 ###
"""
这里放原文。
"""
```

分隔符的作用是减少混淆。模型更容易判断哪些是任务说明，哪些是待处理内容，哪些是输出约束。

## 说清楚“要做什么”

很多时候，与其说“不要做什么”，不如直接说“要做什么”。

比如：

```text
不要写得太复杂。
```

不如改成：

```text
请用 2-3 句话向高中生解释这个概念。
```

“不要复杂”仍然模糊；“2-3 句话”“高中生”会更具体，模型更容易执行。

当然，禁止项不是不能写。涉及安全、格式、边界时仍然可以写“不要”。但 prompt 的主体最好是正向指令，让模型知道该朝哪个方向生成。

## 常见误区

- 误区一：以为 prompt 有标准答案。  
  prompt 更像调参和实验，不同模型、不同任务、不同上下文下都可能需要调整。

- 误区二：一开始就写得很复杂。  
  复杂 prompt 不一定更好。先从简单版本开始，再观察哪里不准，再补哪里。

- 误区三：指令看似丰富，其实很含糊。  
  “专业一点”“高级一点”“不要太啰嗦”都不够明确，最好转成可执行标准。

- 误区四：只靠禁令控制输出。  
  只说不要做什么，模型仍然不知道应该做什么。正向目标通常更重要。

- 误区五：忽略格式对模型的影响。  
  Markdown 标题、列表、代码块、分隔符不是装饰，它们会影响模型理解 prompt 的结构。
