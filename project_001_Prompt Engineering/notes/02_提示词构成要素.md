## Prompt 的典型构成

### 角色

角色是给 AI 定义一个最匹配任务的问题视角。

比如：

```text
你是一位 Python 编程大师。
你是一位严谨的产品经理。
你是一位擅长给初学者解释概念的 AI 应用开发老师。
```

角色的作用不是“表演身份”，而是把问题域先收窄。大模型对 prompt 开头和结尾的内容通常更敏感，所以把角色放在开头，相当于先告诉模型：接下来请用哪个专业视角理解任务。

### 指示

指示是对任务本身的描述，也就是“你要模型做什么”。

比如：

```text
请把下面这段文本分类为中性、否定或肯定。
请根据以下需求生成一个 FastAPI 示例。
请把这段笔记整理成 Obsidian 风格的知识卡片。
```

指示要尽量具体。只说“帮我优化一下”很容易让模型自由发挥；说清楚“优化为更适合初学者复习的笔记”，输出就会稳定很多。

### 上下文

上下文是与任务相关的背景信息。

它可以是项目背景、用户目标、前文对话、业务限制、已有代码、已有风格要求，也可以是多轮交互中前面已经确认过的规则。

上下文的作用是减少模型猜测。很多时候输出不准，不是模型不会，而是它不知道当前任务处在什么场景里。

### 例子

例子是给模型看的参考样本。

在学术表达里，这常和 `one-shot learning`、`few-shot learning`、`in-context learning` 相关：不是重新训练模型，而是在 prompt 里临时给它几个输入输出示例，让它模仿任务模式。

例子对输出正确性帮助很大，尤其适合分类、抽取、格式转换、风格迁移这类任务。

### 输入

输入是任务要处理的具体内容。

最好在 prompt 中明确标识输入区域，比如：

```text
输入文本：
"""
我觉得食物还可以。
"""
```

明确标识输入的好处是避免模型把任务说明、背景信息和待处理内容混在一起。对于代码、长文、JSON、日志这类内容尤其重要。

### 输出

输出是对结果格式的要求。

如果输出要给后续模块自动解析，就应该明确格式，比如 `JSON`、`XML`、Markdown 表格、固定字段列表等。

比如：

```text
请只输出 JSON：
{
  "sentiment": "positive | neutral | negative",
  "reason": "一句话原因"
}
```

输出格式越明确，后续自动化越容易接上。尤其在 Agent、LangChain、工作流节点里，输出不是给人随便看，而是要给程序继续处理。

## 一个可参考的结构

```text
角色：你是谁，用什么视角处理任务

指示：你要完成什么任务

上下文：任务背景、限制、已有信息

例子：给出 1-3 个输入输出示例

输入：明确标识待处理内容

输出：指定返回格式和字段
```

这不是必须照抄的模板。更好的理解是：每次写 prompt 时，都检查一下这六类信息里哪些对当前任务有帮助。

简单任务可能只需要“指示 + 输入”；复杂任务才需要角色、上下文、例子和严格输出格式。

## 常见误区

- 误区一：把模板当成必须遵守的格式。  
  模板只是检查清单。任务简单时，强行套完整模板反而显得啰嗦。

- 误区二：角色写得很响亮，但任务指示很模糊。  
  “你是一位专家”不能替代清楚的任务描述。角色只是收窄视角，真正决定输出方向的是指示和上下文。

- 误区三：没有明确区分输入内容。  
  当 prompt 里混着说明、背景和待处理文本时，模型容易误判哪些内容需要执行，哪些内容只是参考。

- 误区四：忽略输出格式。  
  如果结果要进入后续程序处理，就不能只说“给我答案”，而要提前定义字段、格式和边界。

- 误区五：不给例子，却期待模型自动理解复杂标准。  
  对于分类、抽取、风格模仿这类任务，少量高质量例子往往比一大段抽象规则更有效。
