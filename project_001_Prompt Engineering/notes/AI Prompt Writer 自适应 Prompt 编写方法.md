## 核心思路

**Prompt 编写的核心不是套用一份固定大模板，而是先识别任务类型和信息缺口，再装配真正会改变模型行为、输出或边界的内容。**

简单任务可能只需要几行自然语言；结构化抽取需要字段契约和异常处理；Agent 任务则必须增加操作范围、停止条件、验证方式和人工确认。它们都要形成完整闭环，但不应该长成同一种结构。

AI Prompt Writer 因此采用一套自适应架构：

```text
范围判断
  -> 编写模式
  -> 需求契约
  -> 澄清门
  -> 类型路由
  -> 结构装配
  -> 质量门禁
  -> 标准交付
```

## Skill 负责什么？

这个 Skill 的交付物是**可以直接使用的 Prompt**，而不是 Prompt 所描述任务的执行结果。

| 属于职责范围 | 不属于职责范围 |
| --- | --- |
| 从零编写 Prompt | 直接写文章、代码或分析报告 |
| 优化和规范化已有 Prompt | 单独讲解 Prompt 理论或论文 |
| 适配模型、编程工具、Agent 或工作流 | 编写项目研究和资料综述 |
| 诊断问题并交付重写后的 Prompt | 图像、视频和音频生成 Prompt |

**先判断用户要的是 Prompt，还是任务结果。** 如果目标不是 Prompt 本身，就不应该强行调用 Prompt 编写流程。

## 八阶段编写流程

| 阶段 | 要解决的问题 | 关键判断 |
| --- | --- | --- |
| 范围判断 | 当前任务是否应该由本 Skill 处理 | 目标产物必须是 Prompt 本身 |
| 编写模式 | 这次是在创建、优化、适配还是诊断 | 只选择一种主模式，保留用户原始意图 |
| 需求契约 | Prompt 需要哪些事实和边界 | 标记已明确、可合理补全、必须确认 |
| 澄清门 | 是否必须向用户提问 | 只有缺失信息会实质改变结果时才问，最多 3 个问题 |
| 类型路由 | 应该使用哪类 Prompt 结构 | 选择一个主类型，并叠加必要的次类型规则 |
| 结构装配 | 哪些模块真正需要进入 Prompt | 保留必需模块，按条件添加其他模块，不留空标题 |
| 质量门禁 | Prompt 是否清晰、完整、安全 | 检查任务、输入、约束、输出、完成标准和专项规则 |
| 标准交付 | 怎样让结果可直接使用和复核 | 交付最终 Prompt，并附带质量报告 |

这套流程的“完整”指判断过程和质量闭环完整，**不代表每个 Prompt 都必须写得很长。**

## 需求契约

开始写 Prompt 前，先从需求中提取九项信息：

| 信息 | 要确认的内容 |
| --- | --- |
| 任务 | 模型需要执行什么动作，主要目标是什么 |
| 目标工具或模型 | Prompt 将在哪个模型、Agent 或平台中运行 |
| 输入 | 模型依据什么内容工作，输入边界在哪里 |
| 输出 | 最终交付物的主形态、结构或字段是什么 |
| 约束 | 必须遵守、禁止执行和不能推断的内容 |
| 上下文 | 会改变模型判断的背景、现状和已知条件 |
| 目标受众 | 输出给谁看，需要怎样的语言和深度 |
| 完成标准 | 怎样判断结果已经完整、正确或可使用 |
| 示例 | 是否需要用样例对齐难以描述的格式或判断模式 |

每一项都要归入一种信息状态：

- **已明确**：用户已经提供，或能从当前上下文直接确定。
- **可合理补全**：不会改变用户意图，可以采用保守默认值，并在质量报告中说明。
- **必须确认**：不同答案会改变工具权限、输出契约、业务目标或风险边界。

## 六类 Prompt 的路由

| 类型 | 主要用途 | 最需要补齐的内容 |
| --- | --- | --- |
| 通用文本 | 生成、总结、分析、改写、问答和比较 | 受众、深度、输出结构、事实依据 |
| 代码任务 | 代码生成、解释、审查和单次修改建议 | 技术上下文、文件范围、兼容要求、验证方式 |
| 结构化输出 | JSON、表格、分类、抽取和数据转换 | 字段、类型、来源、枚举、缺失值和异常处理 |
| Agent | 使用文件、命令、浏览器或工具改变外部状态 | 允许范围、禁止动作、检查点、停止条件、人工确认 |
| 自动化工作流 | 定义触发、节点、字段流转和失败处理 | 触发条件、字段映射、重试、幂等和最终产物 |
| 长期角色 | 建立长期复用的角色和固定交互方式 | 职责边界、版本、交互规则、工具能力和初始化 |

类型重叠时，不是简单二选一。应选择一个主类型决定整体结构，再把次类型中不可缺少的规则叠加进去。例如：

- 长期运行的 Agent，以长期角色为主结构，同时加入 Agent 的权限和停止条件。
- 需要输出固定 JSON 的代码任务，以代码任务组织上下文，同时加入结构化输出契约。
- 自动化平台中的 Agent，根据主要执行方式选择主类型，同时补齐另一类的失败处理或操作边界。

## 必需模块与条件模块

所有 Prompt 都要形成最小闭环：

| 必需模块 | 作用 |
| --- | --- |
| 任务 | 定义模型要完成的主动作 |
| 输入 | 定义模型依据什么工作 |
| 约束 | 定义行为、内容和权限边界 |
| 输出 | 定义唯一主交付形态 |
| 完成标准 | 定义什么时候算真正完成 |

角色、背景、技能、工作流、示例、版本和初始化属于条件模块。**只有缺少该模块会改变结果时才加入。**

例如，角色并不是每段 Prompt 的必选开头。一个明确的文本改写任务可能不需要角色；但长期专业助手如果没有职责、边界和交互方式，就很容易在多轮对话中逐渐偏离目标。

## 9 条条件硬标准

这 9 条标准不是同时套用的固定模板，而是按任务条件启用的质量规则。

| 标准 | 适用条件 | 通过要求 |
| --- | --- | --- |
| 简介可追踪版本变化 | 长期角色、生产工作流、多人维护 | 写清输入、输出、关键变化和兼容影响 |
| 输出形态只有一个 | 需要稳定交付或下游读取 | 只指定一个主输出形态，辅助说明不能破坏它 |
| 字段契约全量闭环 | JSON、表格、分类和抽取 | 字段名称、类型、必填性、来源和样例一致 |
| 缺失值策略统一 | 输入可能缺字段或无法识别 | 明确返回 `null`、空数组或错误对象，不编造默认值 |
| 枚举值可校验 | 存在分类或状态字段 | 给出完整允许值和无法判断时的回退策略 |
| 字段来源有约束 | 抽取、识别和事实归纳 | 区分必须来自输入、允许推断和禁止生成的字段 |
| 规则采用“条件 + 结果” | 存在分支、异常或边界情况 | 写成“若 X，则 Y”，不只写抽象要求 |
| 样例覆盖关键路径 | 生产级、结构复杂或错误成本高 | 覆盖正常情况，必要时补充缺失、冲突和非目标输入 |
| 初始化短且能开工 | 长期角色或持续对话 | 只说明身份、输入方式和开始动作，不重复完整规则 |

**标准不适用时应直接跳过，而不是为了形式完整把它强行写进 Prompt。**

## 质量门禁与质量报告

最终 Prompt 需要检查八个维度：任务目标、输入与上下文、约束条件、输出契约、完成标准、类型专项规则、歧义与假设、精简与安全。

质量报告只使用三种状态：

| 状态 | 含义 |
| --- | --- |
| 通过 | 用户已明确提供，或当前上下文能够直接确定 |
| 已补全 | Skill 在不改变用户意图的前提下补入了实质内容 |
| 需确认 | 仍有高影响信息未知，或保留了待补充占位符 |

修改依据必须说明具体做了什么，例如“把‘分析一下’改成‘提取三个主要问题并按严重程度排序’”，不能只写“已优化”或“结构清晰”。

## 安全与表达边界

- **敏感信息只使用占位符**：API key、token、cookie、连接字符串和账号凭据不能进入 Prompt。
- **外部动作需要确认**：删除、发布、发送、购买、提交表单、修改依赖和数据库结构应加入人工确认条件。
- **不索取隐藏思维过程**：需要提高可靠性时，要求验证、列出依据或输出简洁步骤，而不是要求完整展示思维链。
- **不加入无效激励**：不默认使用小费、奖励、威胁、惩罚或情绪操控语句。
- **不假设不存在的工具能力**：普通对话模型不能被默认当作拥有文件系统、终端或浏览器的 Agent。

## 一个实用的编写顺序

1. 判断用户要的是 Prompt 还是任务结果。
2. 确认主任务、运行工具、输入、输出和完成标准。
3. 只对会改变结果的缺失信息提问。
4. 选择主类型，再叠加必要的次类型规则。
5. 先写任务、输入、约束、输出和完成标准。
6. 按条件补充角色、背景、工作流、示例、版本和初始化。
7. 应用当前类型需要的硬标准与安全边界。
8. 删除所有不改变行为、输出或边界的句子。
9. 交付可复制 Prompt，并用质量报告说明补全项和待确认项。

**最终判断标准不是 Prompt 看起来多专业，而是模型是否少猜、下游是否能用、风险是否有边界、结果是否能验证。**
