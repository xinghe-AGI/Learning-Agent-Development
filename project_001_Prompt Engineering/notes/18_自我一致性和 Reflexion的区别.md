# 自我一致性和 Reflexion 的区别

## 它们的核心关系

**自我一致性和 Reflexion 都会多次尝试同一个任务，但前者通过并行采样和投票选择答案，后者通过串行反馈和记忆改变下一次策略。**

```text
自我一致性：
多条独立推理链 -> 聚合最终答案

Reflexion：
第 1 次尝试 -> 反馈 -> 反思记忆
                         -> 第 2 次尝试
```

自我一致性关心“哪些答案重复出现”，Reflexion 关心“上一次为什么失败，以及下一次要怎样改变”。

## 它们的主要区别

| 对比维度 | 自我一致性 | Reflexion |
| --- | --- | --- |
| 尝试关系 | 多次尝试相互独立 | 后一次读取前一次反馈和反思 |
| 执行方式 | 通常并行采样 | 通常串行迭代 |
| 核心机制 | 答案聚合或多数投票 | 反馈、反思和记忆 |
| 是否改变策略 | 不要求改变，只比较结果 | 必须根据反馈调整下一次策略 |
| 适合结果 | 数值、选项、标签等可聚合答案 | 代码、Agent 操作、规划等可重试任务 |
| 依赖条件 | 能提取和归一化最终答案 | 有可靠反馈源和下一次尝试机会 |
| 主要风险 | 多数路径共同犯错 | 错误反馈或低质量反思污染记忆 |
| 成本来源 | 多次模型采样 | 多轮执行、评估和记忆管理 |

**最关键的区别是：自我一致性不要求模型从失败中学习，Reflexion 必须让下一次行动因为反馈而改变。**

## 什么时候用自我一致性？

当任务有相对明确的正确答案，单次推理容易受随机性影响，并且答案可以被统一提取时，优先使用自我一致性。

典型场景包括：

- 数学和逻辑推理。
- 选择题、分类和短答案。
- 多条合理推理路径可能收敛到同一结果。
- 多次采样可以并行完成，不依赖前一次输出。

它解决的是结果稳定性问题，不负责解释每次错误的根因。

## 什么时候用 Reflexion？

**当任务可以重试，而且每次尝试后能获得测试、评分、环境结果或人工反馈时，使用 Reflexion。**

典型场景包括：

- 代码修改后运行测试，再根据失败信息继续修复。
- Agent 执行任务后收到环境奖励或错误状态。
- 长流程任务反复出现同类操作错误。
- 上一次经验能够帮助下一次调整工具、顺序或策略。

Reflexion 的价值不在于重复尝试，而在于把反馈改写成下一次可执行的策略。

## Reflexion 不一定总比自我一致性好

- **没有反馈时无法有效反思**：模型只能生成听起来合理的自我评价。
- **反馈错误会污染记忆**：下一次可能比上一次更差。
- **任务不允许重试时收益有限**：反思没有机会影响后续行动。
- **简单可投票问题没必要维护记忆**：直接采样聚合更轻量。

反过来，如果答案没有统一标准、任务必须根据失败原因修改策略，自我一致性的多数投票也不够用。

## 它们可以怎样组合？

两者可以在不同层级配合：一次尝试内部使用自我一致性提高推理稳定性；整次尝试失败后，再使用 Reflexion 总结经验并调整下一轮策略。

```text
一次尝试：多条推理链 -> 自我一致性聚合 -> 执行动作
任务反馈：失败 -> Reflexion 生成经验 -> 下一次尝试
```

## 一个实用决策顺序

1. 判断最终答案是否可以投票或聚合。
2. 可以聚合且多次结果波动时，使用自我一致性。
3. 判断任务是否有可靠反馈和重试机会。
4. 可以根据反馈改变策略时，使用 Reflexion。
5. 同时存在两类问题时，在单次推理层使用自我一致性，在跨次学习层使用 Reflexion。

**一句话判断：想从多个候选里选答案，用自我一致性；想从上一次失败里学经验，用 Reflexion。**

## 容易混淆的点

- 多次尝试不自动等于 Reflexion，必须存在反馈、反思和策略变化。
- 自我一致性不保留跨次经验，只聚合当前问题的答案。
- Reflexion 的反思不是最终答案，而是下一次尝试的上下文。
- 两者都不能替代真实测试、事实核验或人工判断。
