## 什么是少样本提示？

**少样本提示是一种在 prompt 中提供少量输入输出示例，让模型模仿这些示例来完成新任务的提示方法**。这里的“少样本”指的是：不重新训练模型，也不微调模型参数，而是**在上下文里临时给模型几个示范，让模型理解任务模式、输出格式和判断标准**。

例如，如果想让模型做情感分类，可以这样写：

```text
文本：这家餐厅太棒了。
情感：正面

文本：这次体验很糟糕。
情感：负面

文本：我觉得这次更新还可以。
情感：
```

模型看到前两个示例后，会更容易判断第三个输入也应该输出某种情感类别。

## 少样本提示的工作原理

少样本提示的核心在于上下文学习。模型并没有真的被重新训练，而是在当前 prompt 的上下文里观察示例，然后把示例中的模式迁移到新的输入上。

具体来说：

- **示例提供任务模式**， 模型会从示例中观察“输入是什么样、输出是什么样、输入和输出之间是什么关系”。这比只写抽象规则更直观，尤其适合分类、抽取、格式转换、风格模仿等任务。
- **示例提供输出格式**， 很多时候示例不只是告诉模型答案，也在告诉模型应该如何排版、使用什么字段、答案长度大概是多少。例如 JSON、表格、固定标签、短句解释，都可以通过示例来约束。
- **示例提供标签空间**， 对分类任务来说，示例会告诉模型可选标签有哪些。例如“正面 / 负面 / 中性”，或者“高 / 中 / 低”。少样本提示的效果会受到标签空间、输入分布和输出格式共同影响。
- **示例帮助模型减少歧义**， 如果用户心中的判断标准比较细，比如“什么算高优先级”“什么算负面反馈”“什么算适合拆成任务”，少量例子往往比一长串抽象规则更容易让模型对齐。

## 少样本提示的构造方式

**少样本 prompt 通常由三部分组成：任务说明、若干示例、新输入**。

一种常见结构是：

```text
任务说明：告诉模型要做什么

示例 1：
输入：...
输出：...

示例 2：
输入：...
输出：...

新输入：
输入：...
输出：
```

比如：

```text
请判断用户反馈的情感倾向，只能回答：正面、负面或中性。

输入：这个功能非常好用，节省了我很多时间。
输出：正面

输入：页面经常卡死，我没法正常完成操作。
输出：负面

输入：整体还行，但加载速度可以再快一点。
输出：
```

好的少样本提示不只是“多放几个例子”，而是要放对例子。示例应该覆盖关键边界，格式要稳定，输入要和真实任务接近。

如果示例格式很乱，模型可能会学到错误的输出习惯。如果示例和新任务差距太远，模型可能会误判当前任务类型。

## 示例数量怎么选

少样本提示里的示例数量没有固定答案。常见说法包括 one-shot、few-shot、3-shot、5-shot 等，本质都是在说 prompt 里给了几个示例。

示例太少时，模型可能没有充分理解任务边界；示例太多时，会消耗更多上下文窗口，也可能引入噪声。

一般可以这样理解：
- one-shot 适合格式很简单、只需要一个示范就能看懂的任务。
- 3-shot 到 5-shot 适合分类、抽取、改写、结构化输出这类任务，可以覆盖几个典型模式。
- 更多示例适合边界复杂的任务，但要注意上下文长度、示例质量和成本。

**示例数量不是越多越好。更关键的是示例是否代表真实任务、是否覆盖容易混淆的情况、是否保持同一输出格式。**

## 少样本提示的应用场景

少样本提示适合那些“光靠一句指令不够明确，但又没有必要微调模型”的任务。

- **文本分类**： 例如情感分类、意图识别、问题优先级分类、内容标签分类。给几个标注示例后，模型更容易学会标签边界。
- **信息抽取**： 例如从文本中抽取人名、地名、时间、产品名、错误原因等。示例可以告诉模型抽取粒度和输出格式。
- **格式转换**： 例如把自然语言改成 JSON、表格、任务列表、FAQ、用户故事等。示例能直接展示目标格式。
- **风格模仿**： 例如把内容改写成某种语气、某类笔记风格或某种说明文风格。少量好示例比抽象描述“写得自然一点”更有效。
- **代码相关任务**： 例如生成测试断言、修复简单代码、把需求转成代码模板。示例可以帮助模型理解输入和输出之间的映射。
- **对话和客服场景**： 给几个“用户问题 -> 回复方式”的示例，可以帮助模型模仿语气、边界和服务标准。

## 少样本提示的优势

- **减少对大量标注数据的依赖**， 少样本提示只需要少量示例，不需要为每个任务准备大规模训练集。
- **不需要修改模型参数**， 它发生在推理阶段，只通过 prompt 影响模型输出。相比微调，使用成本更低，迭代也更快。
- **输出更稳定**， 示例能帮助模型理解格式、标签和判断边界，减少零样本提示中常见的自由发挥。
- **适合快速探索任务**， 当还不确定任务是否值得系统化开发时，可以先用少样本提示验证可行性。
- **可以和检索结合**， 在实际系统里，可以从示例库中检索和当前输入最相似的示例，再拼进 prompt。这样比固定放几个示例更灵活。

## 少样本提示的局限性

- **依赖示例质量**， 示例选得不好，模型会被带偏。错误示例、不一致格式、边界不清的样例，都会影响输出。
- **占用上下文窗口**， 示例越多，prompt 越长，成本越高，也会挤占真正输入内容的空间。
- **泛化能力有限**， 少样本提示对示例附近的任务往往有效，但遇到差异很大的输入时，模型不一定能稳定泛化。
- **不擅长复杂多步推理**， 标准少样本提示可以教模型格式和模式，但对于需要多步计算、逻辑推理的问题，单纯加示例不一定够。少样本提示在某些算术推理任务上仍然可能失败，这时需要把中间推理过程显式写出来，也就是升级到思维链提示。
- **可能引入表面模仿**， 模型有时会模仿示例的表面格式，却没有真正掌握背后的判断逻辑。尤其当示例数量少、任务边界复杂时，这个问题更明显。

## 少样本提示的使用经验

少样本提示的重点不是“多给例子”，而是“给模型真正有参考价值的例子”。

写少样本 prompt 时，可以先检查几个问题：

- 示例的输入和真实任务是否相似？
- 示例是否覆盖了容易混淆的边界？
- 所有示例的输出格式是否一致？
- 示例数量是否足够，但没有挤占太多上下文？
- 新输入是否和示例之间有清晰分隔？

如果模型输出不稳定，优先检查示例质量，而不是盲目增加示例数量。很多时候，换一个更贴近任务边界的例子，比多加三个普通例子更有用。

**少样本提示最适合解决“模型大概会做，但不知道你想要哪种答案”的问题。它不是让模型学会全新能力，而是在当前上下文里给模型一个可模仿的任务样式。**

## 零样本和少样本提示的关系

少样本提示和零样本提示的区别在于：零样本只给任务说明，不给示例；少样本会给几个示例，让模型通过上下文学习任务规律。少样本提示常用于零样本效果不稳定，或者任务标准比较隐含、输出格式需要严格模仿的场景。
