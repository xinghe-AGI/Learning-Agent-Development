## 它们的核心关系

**零样本提示和少样本提示是一组递进关系：先用零样本提示测试模型能不能直接理解任务，再用少样本提示通过示例补充任务模式。**

两者都不修改模型参数，也不需要重新训练模型。它们都发生在推理阶段，本质上都是通过 prompt 改变模型的上下文，让模型生成更符合任务目标的答案。

**区别在于：零样本提示只给任务说明和输入；少样本提示会在 prompt 中额外放入几个输入输出示例，让模型模仿这些示例处理新输入。**

可以这样理解：

- **零样本提示**：像是直接对一个人说：“请帮我判断这段话的情感。”
- **少样本提示**：像是先给这个人看几道已经做好的题：“这类句子算正面，这类句子算负面，这类句子算中性；现在请你判断新句子。”

## 它们的主要区别

| 对比维度 | 零样本提示 | 少样本提示 |
|---|---|---|
| 是否提供示例 | 不提供示例 | 提供少量输入输出示例 |
| 依赖什么 | 依赖模型已有知识和指令理解能力 | 依赖模型已有能力，也依赖示例质量 |
| 成本 | prompt 更短，token 成本低 | prompt 更长，token 成本更高 |
| 适合任务 | 常见任务、标准清楚的任务 | 边界隐含、格式严格、需要模仿的任务 |
| 输出稳定性 | 更容易受措辞影响 | 通常更稳定，但会受示例影响 |
| 风险 | 模型可能误解任务或自由发挥 | 示例不好会把模型带偏 |

**最关键的区别不是“哪个更高级”，而是“当前任务需不需要示例帮助模型对齐标准”。**

## 什么时候先用零样本提示？

**零样本提示适合作为第一步。因为它最简单，也最便宜。**

当任务本身比较常见，类别或输出要求也很清楚时，可以先用零样本提示。

比如：

```text
请判断下面用户反馈的情感倾向，只能回答：正面、负面或中性。

用户反馈：
"""
整体还行，但加载速度有点慢。
"""

情感：
```

如果模型能稳定给出合理答案，就没有必要立刻加入示例。

零样本提示尤其适合：

- **文本摘要**， 因为“总结下面内容”是模型很熟悉的任务。
- **简单分类**， 只要类别含义清楚，模型通常可以直接完成。
- **常识问答**， 模型可以依靠预训练知识回答。
- **快速验证**， 想先看模型对任务的大致表现时，零样本更轻。

## 什么时候升级为少样本提示？

**当零样本提示的输出不稳定，或者模型总是误解你的标准时，可以升级为少样本提示。**

常见触发信号包括：

- **输出格式不稳定**， 有时输出一句话，有时输出列表，有时多解释很多内容。
- **分类边界不清**， 模型不知道某些模糊案例应该归到哪一类。
- **风格不符合预期**， 只说“自然一点”“像学习笔记”仍然不够明确。
- **任务有隐含规则**， 比如什么叫“高优先级”、什么叫“可拆分任务”、什么叫“适合复习的知识卡片”。

这时少样本提示可以把抽象标准变成具体样例，让模型有东西可以模仿。

例如：

```text
请判断用户反馈的情感倾向，只能回答：正面、负面或中性。

输入：这个功能非常好用，节省了我很多时间。
输出：正面

输入：页面经常卡死，我没法正常完成操作。
输出：负面

输入：整体还行，但加载速度可以再快一点。
输出：中性

输入：新版本终于解决了我之前的问题。
输出：
```

**这里的示例不只是告诉模型“有哪些标签”，还在告诉模型“什么样的句子对应什么标签”。**

## 少样本不一定总比零样本好

**少样本提示经常比零样本更稳定，但它不是无脑升级。**

- **示例质量不好**，模型可能学到错误模式。比如示例标签错了、格式不一致、示例和真实任务不相似，都会把模型带偏。

- **任务很简单**，少样本提示反而可能显得累赘。它会占用更多上下文窗口，增加 token 成本，也增加维护成本。

- **模型已经通过零样本稳定完成任务**，继续加示例不一定会提升效果。更糟糕的是，示例可能让模型过度模仿某种表面格式，反而降低泛化能力。

所以判断标准不是“少样本更强”，而是：

- 零样本是否已经足够稳定？
- 任务边界是否需要用例子说明？
- 输出格式是否需要模型严格模仿？
- 示例是否足够准确、代表性强、格式一致？

## 它们在学习路径里的位置

在提示工程学习里，可以把零样本和少样本看成最基础的两层能力。

**第一层是零样本提示**：学会把任务、输入、输出格式说清楚。它训练的是“清晰表达任务”的能力。

**第二层是少样本提示**：学会选择示例、排列示例、控制格式和覆盖边界。它训练的是“用样例对齐模型行为”的能力。

再往后，链式思考、自洽性、ReAct、Prompt Chaining 等技术，很多时候都是在这两层之上继续增强。

例如，少样本提示可以给普通输入输出示例，也可以给带推理步骤的示例；后者就接近少样本 CoT。自洽性也可以在零样本或少样本 prompt 上多次采样，再用投票方式提高稳定性。

## 一个实用决策顺序

实际做 prompt 时，可以按这个顺序判断：

- **先写零样本 prompt**，确认模型是否理解任务。
- **如果结果不稳定**，先补充任务说明、类别定义、输入边界和输出格式。
- **如果仍然不稳定**，再加入少量高质量示例。
- **如果任务需要多步推理**，再考虑思维链。
- **如果多次推理结果不一致**，再考虑自洽性。
- **如果任务可以拆成多个环节**，再考虑链式提示。

**这个顺序的好处是：先用最低成本方法建立基线，再根据错误类型逐步升级，而不是一开始就把 prompt 写得又长又复杂。**

## 容易混淆的点

- **少样本提示不是微调**。它只是把示例放进 prompt，没有更新模型参数。

- **零样本提示不是“不提供任何背景”**。它只是“不提供示例”。类别定义、任务背景、输出格式仍然可以写。

- **少样本提示不是示例越多越好**。示例要少而准，最好覆盖关键边界。

- **零样本失败不一定立刻加示例**。也可能只是指令不清楚、输出格式没定义、输入边界没标出来。

- **少样本提示可以提高格式和边界稳定性，但不能保证复杂推理一定正确**。遇到数学、逻辑、多步规划问题时，可能还需要思维链等推理增强方法。
