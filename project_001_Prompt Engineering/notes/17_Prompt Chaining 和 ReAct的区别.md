# Prompt Chaining 和 ReAct 的区别

## 它们的核心关系

**Prompt Chaining 和 ReAct 都能组织多步骤任务，但控制下一步的主体不同：Prompt Chaining 由开发者提前编排流程，ReAct 由 Agent 根据 Observation 动态决定下一步行动。**

可以把它们理解成两种执行方式：

```text
Prompt Chaining：步骤 A -> 步骤 B -> 步骤 C
                 路线通常提前确定

ReAct：Thought -> Action -> Observation
                   -> 根据观察决定下一步
```

## 它们的主要区别

| 对比维度 | Prompt Chaining | ReAct |
| --- | --- | --- |
| 流程控制者 | 开发者或工作流 | Agent |
| 下一步来源 | 预定义节点和条件分支 | 当前推理与环境观察 |
| 流程稳定性 | 高，路径容易预测 | 较低，路径随任务状态变化 |
| 主要输入 | 上一步的结构化中间结果 | 工具返回的 Observation |
| 适合任务 | 摘要、抽取、生成、校验等稳定流水线 | 搜索、排障、网页操作等动态任务 |
| 调试方式 | 检查具体流程节点 | 检查 Thought、Action、Observation 轨迹 |
| 安全重点 | 字段契约、节点失败和重试 | 工具权限、行动确认和停止条件 |
| 成本特点 | 调用次数通常可以提前估算 | 行动轮数不固定，需要限制上限 |

**最关键的区别是：Prompt Chaining 先确定流程再执行，ReAct 在执行中决定流程。**

## 什么时候先用 Prompt Chaining？

当任务阶段明确、输入输出可以提前定义，并且希望流程稳定可复现时，优先使用 Prompt Chaining。

典型场景包括：

- 提取资料 -> 生成摘要 -> 检查格式。
- 生成代码 -> 生成测试 -> 校验结果。
- 文档分类 -> 字段抽取 -> 写入固定结构。
- 每一步需要单独记录、重试或替换模型。

Prompt Chaining 的优势是可观察、可测试和容易限制范围。只要流程路径可以提前描述，就没有必要把控制权完全交给 Agent。

## 什么时候升级为 ReAct？

**当下一步必须根据实时工具结果决定，无法在执行前完整列出路线时，再使用 ReAct。**

常见触发信号包括：

- 搜索结果决定下一步搜索什么。
- 运行命令后要根据报错选择修复动作。
- 页面状态、文件内容或外部环境不断变化。
- 同一目标在不同任务中需要不同工具顺序。

ReAct 用更高自主性换取动态适应能力，因此必须同时增加权限、最大轮数、停止条件和关键动作确认。

## ReAct 不一定总比 Prompt Chaining 好

- **稳定流程不需要动态规划**：固定任务使用 ReAct 会增加随机性。
- **行动成本更难预测**：Agent 可能调用过多工具或重复尝试。
- **调试更复杂**：问题可能来自推理、工具选择、参数或 Observation 误读。
- **安全风险更高**：Agent 能改变外部状态时，需要更严格的授权边界。

能用明确流程解决的问题，优先保持确定性；只有真正依赖环境反馈时，才开放动态决策。

## 它们可以怎样组合？

Prompt Chaining 可以负责外层稳定流程，ReAct 负责其中一个动态节点：

```text
提取任务目标
  -> ReAct 动态搜索和收集证据
  -> 固定校验节点
  -> 固定输出节点
```

这种组合既保留整体可控性，也允许局部任务根据环境调整。

## 一个实用决策顺序

1. 先画出理想任务流程。
2. 能提前确定的步骤使用 Prompt Chaining。
3. 标出必须根据工具结果临场选择的节点。
4. 只在这些动态节点中使用 ReAct。
5. 为 ReAct 节点设置工具白名单、最大轮数和人工确认条件。

**一句话判断：路径已知，用 Prompt Chaining；路径要边执行边发现，用 ReAct。**

## 容易混淆的点

- Prompt Chaining 可以包含条件分支，但分支规则仍由开发者预先定义。
- ReAct 也会形成行动链，但链路不是事先完全固定的。
- 调用多个工具不一定就是 ReAct，固定顺序调用仍可能只是工作流。
- ReAct 可以作为 Prompt Chain 中的一个节点，而不必控制整个系统。
