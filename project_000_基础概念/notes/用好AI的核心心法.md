# 用好 AI 的核心心法

用好 AI 不是寻找一句万能 Prompt，而是建立一套稳定的协作方式：**知道模型的边界，把任务拆清楚，提供足够上下文，用反馈迭代，并对结果负责。**

## 心法一：把模型当作概率生成器，不是事实数据库

LLM 会根据上下文生成高概率的后续内容。它可以整理资料、生成方案和辅助推理，但也可能把不确定内容写得非常流畅。

需要事实准确时：

- 提供可信资料并要求基于资料回答。
- 使用检索或搜索补充时效性信息。
- 要求标注来源、未知项和不确定性。
- 对关键结论进行独立核验。

**模型的语言自信不能代替证据。**

## 心法二：把复杂任务拆成可检查的步骤

“写一份完整商业计划书”很难判断哪里出错；“先确定目标用户，再列假设，再寻找证据，再形成方案”更容易控制。

拆解时确保每一步都有：

- 明确输入。
- 单一目标。
- 可检查输出。
- 失败后的处理方式。

任务变复杂后，可以把多轮对话升级为 Prompt Chaining、工作流或 Agent，但仍要保留每一步的评估点。

## 心法三：用示例降低歧义

“写得专业一点”很模糊；一个输入与输出示例能直接展示格式、风格和判断模式。示例尤其适合：

- 分类标签容易混淆。
- 输出格式必须稳定。
- 风格很难只用形容词描述。
- 任务包含隐性的判断标准。

**示例不是越多越好。** 要优先选择覆盖典型情况和边界情况的高质量示例，避免把错误模式也教给模型。

## 心法四：用反馈迭代，不追求一次完美

第一次输出可以用来暴露理解差异。后续反馈要具体指出：

- 哪一部分不正确。
- 为什么不符合目标。
- 应保留什么。
- 下一版需要怎样改变。

对重复任务，不要只靠聊天记忆。应把有效改动沉淀到 Prompt 模板、示例、评估样本和自动化检查中。

## 心法五：先评估问题类型，再选择技术

| 问题来源 | 优先手段 |
| --- | --- |
| 任务没有说清楚 | 改进 Prompt 和输入边界 |
| 缺少私有或时效知识 | RAG、搜索或明确提供资料 |
| 需要计算或操作外部系统 | 工具调用、代码执行、API |
| 步骤多且需要状态 | 工作流、LangGraph 或其他编排方式 |
| 行为稳定性长期不足 | 在有高质量数据和评估后考虑微调 |

**技术升级应由错误分析驱动，不要因为某个词热门就先搭复杂架构。**

## 心法六：始终保留验证与责任边界

- 高风险任务必须有人类专业判断。
- 自动化执行前要限制权限和操作范围。
- 记录输入、输出、工具调用和错误，便于回溯。
- 用测试集比较版本，避免只凭单次好结果判断。
- 把“模型说了什么”和“系统实际做了什么”分开。

## 一套最小协作循环

```text
明确目标
  -> 提供上下文与示例
  -> 生成候选结果
  -> 核验事实与格式
  -> 给出具体反馈
  -> 沉淀模板和测试
```
