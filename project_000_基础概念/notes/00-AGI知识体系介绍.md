## AI 是什么？

**人工智能（AI）是让机器完成通常需要感知、推理、学习、规划或语言能力才能完成的任务的一组技术与方法。**

AI 不只等于机器学习。规则系统、搜索与规划属于经典 AI；机器学习和神经网络则是现代 AI 的主流实现方式。两者更详细的关系见[规则、搜索与机器学习都属于 AI 吗？](<为什么机器学习、神经网络是AI，而规则搜索不是？.md>)。

## AGI 是什么？

**AGI（Artificial General Intelligence，通用人工智能）通常指能够跨领域学习、迁移和完成广泛任务的人工智能。** 目前没有一个被所有研究者共同接受的精确定义，也没有统一的“已经实现 AGI”的判定标准。

### 狭义 AI 与 AGI

| 维度 | 狭义 AI（ANI） | 通用人工智能（AGI） |
| --- | --- | --- |
| 任务范围 | 在限定任务上表现突出 | 能跨任务迁移并学习新能力 |
| 能力来源 | 针对任务设计或训练 | 需要更广泛、可组合的通用能力 |
| 评估方式 | 用明确任务指标评估 | 需要跨领域、适应性和安全性评估 |
| 当前状态 | 已广泛应用 | 定义和实现状态仍有争议 |

![狭义 AI 与 AGI 的能力范围](<pictures/agi-xinghe-v2-02-ani-vs-agi.png>)

图中的“尚不存在”应理解为：**尚无被普遍认可、通过统一标准验证的 AGI 系统**，而不是对所有研究观点下绝对结论。

### 讨论 AGI 时常见的能力维度

| 能力 | 关注点 |
| --- | --- |
| 感知 | 处理文本、图像、声音和其他环境信号 |
| 记忆 | 在当前任务和长期交互中保留、检索信息 |
| 推理 | 组合信息、形成计划、检查一致性并修正错误 |
| 元认知 | 评估自身状态、能力边界与结果可靠性 |
| 行动 | 调用工具、影响环境并根据反馈调整策略 |
| 对齐与安全 | 在目标、约束和人类价值边界内行动 |

## LLM 在这张地图中的位置

**LLM（Large Language Model，大语言模型）是深度学习模型的一类，擅长处理和生成语言及其他序列化信息。** LLM 可以成为通用智能系统的重要组件，但 LLM 与 AGI 不能直接画等号。

LLM 的基础关系可以压缩为：

```text
Prompt
  -> Tokenization
  -> Transformer 根据上下文计算下一个 Token 的概率
  -> 选择一个 Token 并追加到上下文
  -> 重复，直到结束
```

对应的概念链路：

- [Token](<Token.md>)：模型读写文本时使用的离散单位。
- [Transformer 架构](<Transformer 架构.md>)：组织模型计算的网络架构。
- [自注意力机制](<自注意力机制.md>)：让每个位置聚合上下文信息。
- [下一个 Token 预测训练目标](<下一个词预测训练目标.md>)：许多生成式 LLM 在预训练阶段使用的目标。
- [自回归生成方式](<自回归生成方式.md>)：推理时逐 Token 生成的方式。
- [Prompt](<Prompt.md>)：为生成过程提供任务、上下文与约束。

## 从模型能力到 AI 应用

AI 进入产品后，人的参与程度和模型的自主程度可以不同：

| 模式 | 人与 AI 的分工 | 典型形态 |
| --- | --- | --- |
| Embedded | AI 嵌入某个固定环节 | 分类、抽取、推荐、语音转写 |
| Copilot | 人主导流程，AI 持续辅助 | 写作助手、编程助手、分析助手 |
| Agent | 系统根据目标规划步骤并调用工具，人负责授权与监督 | 调研、任务编排、跨系统操作 |

![AI 应用模式](<pictures/agi-xinghe-v2-07-ai-application-modes.png>)

这三种模式不是严格的产品分类，更像一条从“局部能力”到“较高自主性”的连续光谱。

## 常见 LLM 应用技术

| 技术 | 主要解决的问题 | 不要误解为 |
| --- | --- | --- |
| Prompt Engineering | 让模型更准确理解任务、上下文和输出约束 | 给模型增加新知识 |
| RAG | 在生成前检索外部知识，并把证据放入上下文 | 模型永久记住检索内容 |
| Function Calling / Tools | 让应用根据模型给出的结构化意图调用外部工具 | 模型自己直接执行所有操作 |
| Fine-tuning | 调整模型在特定任务、风格或行为上的参数分布 | 一种通用、低成本的知识库替代方案 |

这些技术可以组合，不是四选一。一个应用常常先用 Prompt 建立基线，再按问题类型增加检索、工具调用、工作流或微调。

## 如何选择技术路线？

1. **先定义任务和评估标准**：输入是什么、输出是什么、怎样才算正确。
2. **先做最低成本基线**：用直接模型调用和清晰 Prompt 验证任务是否可行。
3. **缺少外部知识时增加 RAG**：尤其是私有、时效性或需要引用来源的信息。
4. **需要真实动作时增加工具调用**：查询数据库、发起 API 请求或操作业务系统。
5. **流程复杂时增加编排与状态管理**：让步骤可恢复、可观察、可人工介入。
6. **问题稳定且有高质量数据时再评估微调**：用实验比较收益、成本与维护负担。

**技术选择的起点不是“哪种方案最先进”，而是“当前误差究竟来自指令、知识、工具、流程还是模型行为”。**
